# Product Definition

> Source: MASTER_SPECIFICATION.md §2

---

# Purpose

This document defines what Kisuke is, what it is not, its scope, and its intended operating boundary.

It defines product boundaries only.

It does not define implementation.

---

# Product Statement

Kisuke is a local-first context reconstruction engine.

Its purpose is to reconstruct working context so interrupted work can be resumed with minimal cognitive effort.

---

# Primary Goal

Reduce context reconstruction time.

Not note-taking.

Not information storage.

Not project management.

---

# Core Promise

> Kisuke reconstructs working context.

It does not simply retrieve information.

---

# Target Users

Kisuke is designed for people who perform long-running knowledge work.

Examples:

- Software Engineers
- AI Engineers
- Students
- Researchers
- Founders
- Technical Writers

---

# Scope

Kisuke manages:

- Missions
- Projects
- Tasks
- Knowledge
- Cookbook
- Decisions
- Meetings
- Reviews
- Resources
- Context

Kisuke references:

- Git repositories
- Documentation
- Websites
- Calendars
- Files
- External services

---

# What Kisuke Is

- Context reconstruction engine
- Personal operating system
- Knowledge workflow engine
- Resume engine
- Decision memory
- Local-first application

---

# What Kisuke Is Not

Kisuke is not:

- Note-taking software
- Search-first software
- Project management software
- Cloud service
- Autonomous AI agent
- Git replacement
- Calendar replacement
- Obsidian replacement
- File manager

---

# Product Boundaries

Kisuke owns:

- Context
- Relationships
- Metadata
- Resume workflow

Kisuke does not own:

- Git history
- Calendar events
- Source code
- External documentation
- Browser bookmarks

Those remain owned by their original systems.

---

# Core Capabilities

- Capture
- Resume
- Search
- Review
- Decision tracking
- Knowledge organization
- Context reconstruction

---

# Success Criteria

A successful Kisuke session should allow a user to:

- Resume work quickly.
- Understand current project state.
- Find the next meaningful action.
- Recover prior decisions.
- Avoid rereading documentation unnecessarily.

---

# Design Principles

- Local-first
- Offline-first core
- Markdown-first
- Git-native
- Single ownership
- Reference over duplication
- Resume before search
- Integrate before rebuild
- AI optional

---

# Out of Scope

The following are intentionally excluded from v1:

- Team collaboration
- Multi-user synchronization
- Real-time editing
- Permissions
- Authentication
- Billing
- Mobile application
- Web application

These may be considered separately in future versions through the RFC process.
